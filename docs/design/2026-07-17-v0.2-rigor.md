# v0.2 "Rigor" — design

- **Date:** 2026-07-17
- **Status:** accepted
- **Scope:** three features, zero new dependencies, ships as 0.2.0

## Context

v0.1.0 shipped a complete command surface, but two gaps separate what Switchyard
promises from what it can guarantee:

1. **The coordination tool is not itself concurrency-safe.** Every mutating
   command does read → modify → write on `.fleet/state.json` with no
   inter-process lock (`src/lib/state.ts`). The write-then-rename protects
   against torn files, not lost updates: two concurrent `fleet spawn`s both read
   the same state, both write, and the last writer wins — one agent silently
   vanishes from state. An agent missing from state is invisible to
   `fleet check`, which is the collision detector failing open. Concurrent
   invocation is the designed usage: agents themselves run `fleet` commands via
   `--json`.
2. **Collision detection is heuristic where git can give ground truth.**
   File-level overlap over-warns by design; `--lines` refines it but carries a
   documented merge-base caveat. Git ≥ 2.38 `git merge-tree --write-tree`
   performs a real in-memory three-way merge and reports exactly which files
   would conflict, touching no worktree and no index.

v0.2 closes both gaps and adds a safety escape hatch (`fleet undo`), making the
core promise — safe parallel agents, safe merges — provable rather than
plausible.

## Goals

- No lost state updates under concurrent `fleet` invocations on one repo.
- `fleet check` and the `fleet merge` gate report *predicted merge conflicts*,
  not just shared files, whenever the installed git supports it.
- The last `fleet merge` is reversible with one command.

## Non-goals (explicitly out of v0.2)

- `fleet sync` conflict pre-simulation.
- Multi-level / history undo (single-level only).
- Config knob for lock timeout (fixed default; add on demand).
- `fleet init` — deferred to v0.3 with the agent-loop features.
- Cross-repo, remote, or submodule anything (unchanged v1 limitations).

## Feature 1 — inter-process lock

New module `src/lib/lock.ts`:

```ts
interface LockInfo {
  pid: number;
  command: string;   // e.g. "spawn", for error messages and doctor
  startedAt: string; // ISO 8601
}

async function withLock<T>(
  repoRoot: string,
  command: string,
  fn: () => Promise<T>,
): Promise<T>;
```

- **Acquire:** ensure `.fleet/` exists, then create `.fleet/lock` with
  `writeFileSync(..., { flag: 'wx' })` — an atomic exclusive create on both
  NTFS and POSIX — containing `LockInfo` as JSON.
- **On `EEXIST`:** read the holder. If `process.kill(pid, 0)` throws `ESRCH`
  the holder is dead: remove the stale lock, log one line about it, retry
  immediately. If the holder is alive (including `EPERM`), retry on a ~100 ms
  delay with small jitter, up to a fixed 10 s total, then throw `FleetError`
  naming the holder's PID, command, and lock age, with a hint that
  `fleet doctor --fix` removes dead locks.
- **Release:** in `finally`, re-read the lock and delete it only if the PID is
  ours — a stale-takeover race must never delete a successor's lock.
- **Wrapped commands (the mutators):** `spawn`, `merge`, `remove`, `clean`,
  `sync`, `doctor --fix`, `undo`. One repo-wide lock, one mutator at a time —
  coarse by design at this scale.
- **Not wrapped:** all read-only commands, and `exec` — it runs long agent
  workloads, serializing it would defeat the tool, and it never writes state.
- **`fleet doctor`:** reports a present lock with holder PID, command, age, and
  liveness; `--fix` removes dead-PID locks (acquisition already self-heals, so
  this is visibility plus manual repair).
- **Accepted, documented limitations:** PID liveness is same-machine only
  (consistent with state being single-machine by design); PID reuse is
  tolerated the same way npm/yarn lockfiles tolerate it.

## Feature 2 — conflict prediction via `git merge-tree`

### Capability probe

`src/lib/git.ts` gains `gitVersion()` (parsed from `git version`, cached
per process) and `supportsMergeTree()` (≥ 2.38). `fleet doctor` surfaces it:
"conflict prediction: available" / "unavailable (git X.Y < 2.38)".

### Check semantics (default when capable)

When git ≥ 2.38, `fleet check` upgrades automatically. For each unordered
*pair* of agents sharing at least one changed file, run from the main repo:

```
git merge-tree --write-tree -z --name-only <branchA> <branchB>
```

a real in-memory merge over the shared object database. Every shared file gets
a verdict, with precedence **conflicts > uncommitted > clean**:

- **conflicts** — some pair's simulation predicts a merge conflict in this
  file. Counted as a collision; exit code 1.
- **uncommitted** — a sharer has uncommitted edits here (or a branch/base was
  missing, so simulation is incomplete). Fail closed: counted as a collision;
  exit code 1.
- **clean** — every sharing pair's committed sides auto-merge. Demoted to an
  informational list (like `--lines` disjoint files today); does not affect
  the exit code.

Output changes:

- Human table gains a VERDICT column; a trailing dim line names the mode
  ("verdicts from git merge-tree simulation" vs "file-level only (git < 2.38)").
- JSON (additive): each collision may carry
  `verdict: 'conflicts' | 'uncommitted'`; `CheckResult` gains
  `cleanMerges?: Collision[]` and `prediction: 'merge-tree' | 'files'` so
  scripts know which semantics ran.
- `--files-only` flag restores exact v0.1 behavior (and is implied on
  git < 2.38). `--lines` stays orthogonal and combinable; when both run, the
  table shows both LINES and VERDICT columns.

Behavior note for the changelog: on capable git, overlapping files that merge
cleanly stop failing `fleet check`. Semantics improve and vary with git
version; acceptable at current adoption, called out prominently.

### Merge gate

`fleet merge` (which reuses `check()`) blocks on verdicts **conflicts** and
**uncommitted**, and no longer blocks on **clean** overlaps. Prediction need
not be infallible to be safe: the existing abort-on-conflict net in
`src/commands/merge.ts` still catches any mis-prediction, so a wrong "clean"
verdict costs an aborted merge, never a broken repo.

### Parsing note

simple-git surfaces failures based on stderr, not exit codes (see the
`branchExists` comment in `src/lib/git.ts`); conflicted `merge-tree` exits 1
with empty stderr and structured stdout. The implementation pins this behavior
down with a test either way, and parses `-z` (NUL-terminated) output: first
record is the written tree OID, subsequent records are conflicted paths.

## Feature 3 — `fleet undo`

Undoes the **last `fleet merge`**, single level.

### Recording (inside `fleet merge`)

Immediately before `git merge` starts (after the gate and `preMerge` hook
pass), write two refs — refs, not just JSON, so the commits stay GC-safe after
branch deletion:

- `refs/fleet/undo-head` → current HEAD of the target branch;
- `refs/fleet/undo-branch` → the agent branch tip.

After the merge **succeeds** (including the cleanup step), write
`.fleet/undo.json`:

```json
{
  "version": 1,
  "agent": {
    "name": "claude",
    "branch": "fleet/claude",
    "baseBranch": "main",
    "worktreePath": ".fleet/worktrees/claude",
    "createdAt": "2026-07-16T09:30:00.000Z"
  },
  "into": "main",
  "headBefore": "<oid>",
  "branchTip": "<oid>",
  "headAfter": "<oid>",
  "cleaned": true,
  "branchDeleted": true,
  "mergedAt": "2026-07-17T12:00:00.000Z"
}
```

If the merge aborts or fails, both refs are deleted and no `undo.json` is
written — `fleet undo` can never act on a failed merge, and a crash mid-merge
leaves at worst orphaned refs with no record (refused by undo, reported by
doctor). A subsequent merge overwrites the record (single level). `state.json`
schema stays v1; undo state lives in its own file.

### Undo (new command, takes the lock)

Refusal conditions, each with a specific error:

- current branch ≠ recorded `into`;
- `HEAD` ≠ recorded `headAfter` ("history moved since the merge");
- main worktree has uncommitted changes;
- no undo record, or a recorded ref is missing.

Then: `git reset --hard` `into` to `headBefore` (safe: the preconditions above
guarantee a clean tree at the expected commit), recreate `fleet/<agent>` at `branchTip`
(only if `branchDeleted`), re-add the worktree at the recorded path (only if
`cleaned`), restore the state entry, delete both refs and `undo.json`. The
`--no-clean` and kept-dirty-worktree merge variants therefore undo correctly:
only what was actually removed is restored.

`fleet doctor` reports a pending undo record (age, agent) as information.

## Testing

All tests in the existing style: real temporary git repositories via
`makeTempRepo()`, no git mocks.

- **Lock:** in-process tests for contention timeout, stale dead-PID takeover
  (write a lock file with a known-dead PID), and release-on-error. Plus one
  true inter-process race: two child processes running the built CLI
  (`node dist/cli.js spawn a` / `spawn b`) against one temp repo, asserting
  both agents end up registered. CI already builds before testing; locally a
  vitest global setup keeps `dist/` fresh for this test.
- **Merge-tree:** temp repos crafting (a) a real conflict — same-line edits on
  two agent branches, (b) a clean same-file merge — disjoint committed hunks,
  (c) an uncommitted overlap; assert all three verdicts, exit codes, JSON
  fields, and the merge gate's block/allow behavior. The `--files-only` /
  old-git fallback path is asserted by flag. The simple-git exit-1 parsing
  behavior gets a dedicated pin-down test.
- **Undo:** merge → undo → assert branch tip, worktree presence, and state
  entry are restored (`rev-parse` equality on refs); each refusal case;
  double-undo refusal; the `--no-clean` variant.
