# v0.3 "Fleet Protocol" — design

- **Date:** 2026-07-19
- **Status:** **partially superseded on scope by [../2026-v0.3-plan.md](../2026-v0.3-plan.md).**
  The MCP transport shipped in 0.3.0, but with **four read-only tools**, not the
  six proposed here — `fleet_claim` and `fleet_spawn` are not exposed. **Claims**
  and **`fleet init`** are deferred to v0.4: both add new persisted state and new
  mutations, which the plan's read-only scope rules out.
  The design reasoning below stands and remains the reference for claims when
  they land; only the release it targets has moved.
- **Scope:** three features, zero new dependencies. The MCP surface shipped as
  0.3.0; claims and `fleet init` did not.

## Context

v0.1 shipped the command surface; v0.2 made its guarantees provable (inter-process
lock, `git merge-tree` conflict prediction, `fleet undo`). Both releases share a
structural assumption: **a human runs every command.**

That assumption is where the remaining value leaks. Switchyard's collision
detector only fires when someone remembers to run `fleet check`, and by then both
agents have already written the conflicting code. Detection is late by
construction. The agents doing the work — the actual subjects of this tool — have
no way to consult it.

v0.3 closes that gap. Switchyard becomes something agents talk to directly, and
gains a primitive for declaring intent *before* the edit rather than discovering
the overlap after it.

## Goals

- An agent can query fleet state mid-task without a human relaying it.
- An agent can declare what it intends to work on, and read what others declared.
- A new repo can be brought into the fleet workflow in one command.
- Zero new runtime dependencies (still exactly `commander`, `simple-git`,
  `chalk`, `cli-table3`).

## Non-goals (explicitly out of v0.3)

- Mutating MCP tools (`spawn`, `merge`, `remove`, `clean`, `sync`, `undo`).
- Enforcing claims — claims are advisory in this release, by design.
- Glob patterns in claims (path prefixes only; globs are a future extension).
- Claim TTLs / expiry (claims are lifecycle-tied to their agent).
- HTTP or SSE MCP transports (stdio only).
- Cross-repo, remote, or submodule anything (unchanged v1 limitations).

## Feature 1 — `fleet mcp`

A hand-rolled MCP server over stdio. No SDK: MCP's core is `initialize`,
`tools/list`, and `tools/call` over newline-delimited JSON-RPC 2.0, which is a
small and stable surface for a read-mostly tool set. Keeping the four-dependency
footprint intact is a stated project value, and the protocol layer is tractable
to own.

### New modules

**`src/lib/jsonrpc.ts`** — JSON-RPC 2.0 over newline-delimited stdio. Reads
stdin line by line, parses, dispatches to a handler map, writes responses.
Malformed input and unknown methods produce well-formed JSON-RPC error objects;
nothing throws out of the loop.

**`src/lib/mcp.ts`** — the MCP layer: `initialize` handshake (protocol version,
server info, capabilities), `tools/list` (tool definitions as plain data with
JSON Schema inputs), `tools/call` (dispatch plus error mapping). Transport-
agnostic so it is unit-testable without a subprocess.

**`src/commands/mcp.ts`** — wires fleet command functions in as tool handlers.
This layer is thin: `list`, `status`, and `check` already return structured
objects to serve `--json`, so their results are the tool results.

### Exposed tools

| Tool | Args | Returns |
| --- | --- | --- |
| `fleet_list` | — | all agents with branch, base, ahead/behind, uncommitted count |
| `fleet_status` | `agent` | one agent in detail |
| `fleet_check` | `lines?`, `filesOnly?` | collisions with merge-tree verdicts, plus clean-merge list |
| `fleet_claims` | — | all claims and their overlaps |
| `fleet_claim` | `agent`, `paths[]`, `note?` | the recorded claim |
| `fleet_unclaim` | `agent`, `paths?` | the remaining claim, if any |

**Deliberately not exposed:** `spawn`, `merge`, `remove`, `clean`, `sync`,
`undo`. An agent that can call `merge` can rewrite branch history; one that can
call `remove --force` can destroy another agent's uncommitted work. Read plus
declare-intent is the safe surface for the first release, and the boundary is
easy to widen later — impossible to narrow after agents depend on it.

### Long-lived process discipline

`fleet mcp` is the first Switchyard command that does not run and exit
immediately. Three consequences, each with a test:

- **Never hold the mutation lock across the process lifetime** — that would
  deadlock every other fleet command. The lock is taken only inside
  `fleet_claim` / `fleet_unclaim`, for the duration of that one write.
- **Re-read state on every call.** Agents spawn, merge, and vanish underneath a
  running server; any cached `state.json` is stale the moment it is read.
- **Survive external mutation.** Branches deleted, worktrees removed, or
  `doctor --fix` run in another terminal must produce structured tool errors,
  never a crash that takes down the calling agent's session.

## Feature 2 — claims

An advisory declaration of intent, readable by every agent.

Claims deliberately do **not** block. Switchyard has no filesystem hooks and
cannot intercept a write, so enforcement would be partly illusory; worse,
work spans minutes to hours, so exclusive leases would wedge the fleet whenever
an agent crashed holding one. The value is informational: an agent asks who is
working in `src/api/` *before* it starts, and routes around.

### Storage

`.fleet/claims.json`, following the precedent v0.2 set with `undo.json` —
separate concern, separate file, `state.json` stays schema v1.

```json
{
  "version": 1,
  "claims": [
    {
      "agent": "alpha",
      "paths": ["src/api/", "src/lib/routes.ts"],
      "note": "refactoring the routes layer",
      "claimedAt": "2026-07-19T12:00:00.000Z"
    }
  ]
}
```

Writes go through the same write-then-rename discipline as `state.json`, under
the mutation lock.

### Path semantics

Claims are **path prefixes**, not globs: a claim is a file path or a directory
path, repo-root-relative with forward slashes. This covers the real case ("I'm
working in `src/api/`") exactly, needs no glob dependency, and avoids
hand-rolling glob matching — a bug-dense surface for little gain here. Globs
remain a clean future extension.

Overlap is **path-component-aware, not string-prefix**. Both paths are split on
`/` and compared component by component; two paths overlap when one's component
list is a prefix of the other's. So `src/api/` overlaps `src/api/routes.ts` but
**not** `src/apifoo.ts` — the naive string comparison gets that case wrong, and
it is exactly the case a real repo hits.

Claim paths are validated on entry with the same rules `copyOnSpawn` already
applies in `src/lib/config.ts`: repo-root-relative only, absolute paths and `..`
segments rejected. Paths need not exist on disk — claiming a file you are about
to create is a legitimate and useful declaration of intent.

### Commands

- `fleet claim <agent> <path...> [--note <text>] [--replace]` — record intent.
  Additive by default; `--replace` overwrites the agent's paths.
- `fleet claims [--json]` — table of all claims with overlap annotations.
- `fleet unclaim <agent> [path...]` — release all paths, or the named ones.

### Overlap kinds

- **claim ↔ claim** — two agents declared intersecting territory. Informational.
- **claim ↔ edit** — an agent has actually changed files under another agent's
  claim. The real signal, and strictly earlier than today's post-hoc detection,
  because it fires while the work is in progress rather than at merge time.

### Lifecycle

A claim dies with its agent: `fleet merge` and `fleet remove` drop the agent's
claims as part of cleanup. Because claims are advisory, a stale claim is noise
rather than a wedge, so no TTL is warranted.

### `fleet check` integration

`fleet check` grows a claims section, printed only when claims exist. **Its exit
code is unchanged** — claims are advisory, and letting them fail CI would make
exit codes depend on bookkeeping hygiene rather than real merge risk. JSON output
gains an additive `claims` field.

## Feature 3 — `fleet init`

One command to bring a repo into the fleet workflow:

- Writes a starter `.fleetrc.json` with the `$schema` key wired up. Refuses to
  clobber an existing config without `--force`.
- Ensures `.fleet/` is in `.git/info/exclude` (today this happens on first
  spawn).
- Appends a fleet protocol block to `AGENTS.md` / `CLAUDE.md` if present —
  telling agents where their worktree is, to claim before editing, and to check
  before merging. Idempotent: re-running never duplicates the block.

This is what makes the rest discoverable. An agent only follows the protocol if
something tells it the protocol exists.

## Testing

All tests in the existing style: real temporary git repositories via
`makeTempRepo()`, no git mocks.

- **`jsonrpc.ts`** — unit tests for framing, malformed JSON, unknown methods,
  and error-object shape. Pure functions, no subprocess.
- **MCP server** — spawn the built CLI as a child process
  (`node dist/cli.js mcp`), speak JSON-RPC over stdin/stdout, and assert the
  `initialize` handshake, `tools/list` contents, and every `tools/call` result.
  Reuses the subprocess pattern and vitest global setup established by v0.2's
  inter-process lock test.
- **Long-lived correctness** — with the server running, spawn a new agent from
  the test, call `fleet_list` again, assert the new agent appears. Proves the
  per-call state re-read rather than asserting it.
- **Lock discipline** — assert `.fleet/lock` is absent while the server idles,
  and that `fleet_claim` acquires and releases it.
- **Claims** — prefix overlap in both directions, claim↔edit detection,
  lifecycle removal on merge and on remove, JSON output shape.
- **`init`** — writes config, is idempotent on re-run, refuses to clobber
  without `--force`, appends the protocol block exactly once.
