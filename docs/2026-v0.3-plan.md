# v0.3 plan — MCP surface + shipped Claude Code skill

- **Date:** 2026-07-19
- **Status:** **shipped in 0.3.0.** Tasks 0–6 all landed as planned; the
  reduced scope held (four read-only tools, no new dependencies, no state
  schema change, `lock.ts` untouched). Task 0's findings are recorded in §2
  below.
- **Baseline:** 0.2.0, live on npm

## Locked decisions

| # | Decision | Effect |
| --- | --- | --- |
| **Q1** | **Same package.** MCP server ships inside `@switchyardhq/switchyard`; no separate package. | `fleet mcp` subcommand on the existing `fleet` bin. `docs/deployment.md` unchanged. |
| **Q2** | **v0.3 is the MCP surface over existing read/report/lock operations only.** `fleet suggest-merge-order` and all graph logic deferred. | Four read-only tools. **No mutating tools at all** — including `fleet_spawn`. No new algorithms. |

### Scope consequences of Q2

Q2's wording — *read/report/lock operations only* — excludes mutations, so
`fleet_spawn` is **out**, not just the merge-ordering graph. Three things follow:

1. **The tool set is four, all read-only.** No tool writes state.
2. **The server never acquires the mutation lock**, which removes the sharpest
   correctness risk in the previous draft (see §1.2). It still *reads* lock state
   via `fleet_lock_status`.
3. **Agents can observe the fleet but not join it.** A human still runs
   `fleet spawn`. This is a genuine product limitation, not a technicality: v0.3
   delivers "agents can see the fleet," not "agents can provision themselves."
   The shipped skill must say so plainly, or agents will hunt for a spawn tool
   and fall back to raw `git worktree`, defeating the point.

Re-adding `fleet_spawn` later is a contained task (~80 lines + lock-discipline
tests) — the boundary is easy to widen and impossible to narrow once agents
depend on it.

### Also deferred to v0.4

**Claims** and **`fleet init`**, from `docs/design/2026-07-19-v0.3-fleet-protocol.md`.
Both introduce new persisted state and new mutations, which Q2
rules out. Claims are also more useful over a transport that already exists.

## 1. Two findings that shape the work

### 1.1 Command functions print to stdout — this blocks MCP outright

MCP's stdio transport uses **stdout as the protocol channel**. Any stray write
corrupts the JSON-RPC stream and breaks the client session.

Switchyard's command functions currently compute *and* print:

```
src/commands/check.ts:211   console.log(JSON.stringify(result, null, 2));
src/commands/check.ts:216   console.log(ok(`No collisions across ...`));
src/commands/doctor.ts:284  console.log(JSON.stringify({ checks, healthy }, ...));
src/commands/list.ts:69     console.log(JSON.stringify(listings, null, 2));
```

So `check()` cannot be called from a tool handler as-is. **The codebase already
contains the correct pattern**, in `src/commands/list.ts`:

| Function | Role |
| --- | --- |
| `collectListings(options)` | pure — "gather the data `fleet list` displays, without printing anything" |
| `buildListTable(listings)` | pure — renders |
| `list(options)` | orchestrates + prints |

Task 1 extends that separation to `check` and `status`. It is a genuine
improvement to the codebase rather than MCP-specific scaffolding, and it is a
**no-behavior-change refactor** guarded by the existing suite.

*Alternative considered and rejected:* redirect `console.log` to stderr during a
tool call (~10 lines). It works — MCP leaves stderr free — but it is a global
monkey-patch that silently swallows output, breaks if any code writes to
`process.stdout` directly, and leaves the tangle in place. Keep it as a
defensive net, not as the mechanism.

### 1.2 `withLock` reentrancy — risk substantially reduced by Q2

`src/lib/lock.ts:151`:

```ts
if (holdDepth > 0) {
  holdDepth += 1;
  try { return await fn(); } finally { holdDepth -= 1; }
}
```

`holdDepth` is **process-global**. Correct for its designed case (merge →
autoClean → clean nested in one command), but unsafe in a long-lived process
where two calls can be in flight — the second would see `holdDepth > 0` and
re-enter *without acquiring the lock*.

**Under Q2 this cannot trigger: no exposed tool calls `withLock`.** The server
only reads.

**Serialize tool handling anyway.** Two reasons: v0.4 reintroduces mutations via
claims, and retrofitting serialization into a server that was built assuming
concurrency is far riskier than building it in now. It costs a request queue and
buys immunity to a bug class that is hard to reproduce and easy to ship.

`lock.ts` is not modified. v0.2's locking stands exactly as shipped.

## 2. MCP surface

### Tool set — four, all read-only

| Tool | Inputs | Output | Maps to | New logic? |
| --- | --- | --- | --- | --- |
| `fleet_list` | — | `AgentListing[]` | `collectListings()` (`list.ts:36`) | **None** — already pure |
| `fleet_status` | `agent: string` | `StatusResult` | `status()` (`status.ts:30`) | Extract pure core (Task 1) |
| `fleet_check` | `lines?: bool`, `filesOnly?: bool` | `CheckResult` | `check()` (`check.ts:59`) | Extract pure core (Task 1) |
| `fleet_lock_status` | — | `LockStatus` | `lockStatus()` (`lock.ts:90`) | **None** — already pure |

`AgentListing`, `StatusResult`, `CheckResult`, and `LockStatus` are existing
exported interfaces. Tool outputs are those objects verbatim — **no new result
shapes and no new algorithms**, satisfying Q2. The `--json` work in v0.1 turns
out to have been the groundwork for this.

Two of four tools need no changes at all; two need only the Task 1 extraction.

### Not exposed

`spawn`, `merge`, `remove`, `clean`, `sync`, `undo` — every mutation, per Q2.

A pleasant side effect: because `spawn` is out, the `postSpawn` hook (arbitrary
shell from `.fleetrc.json`) is **not reachable from an agent** in v0.3. The
previous draft had to flag that as a newly-widened trust boundary; it now
disappears. Re-adding `fleet_spawn` in v0.4 reopens it and will need the note.

### Protocol details requiring verification — RESOLVED 2026-07-19

Verified against the published specification before Task 2, as required. All six
answers, and what each one changed:

1. **Protocol version string:** `2025-11-25` is the current revision. A
   `2026-07-28` revision is in release-candidate state and makes MCP stateless at
   the protocol layer, but clients speaking it fall back to the `initialize`
   handshake against older servers, so targeting `2025-11-25` stays
   interoperable. Revisit once that revision is final.
2. **`initialize` result shape:** `{ protocolVersion, capabilities, serverInfo,
   instructions? }`. `capabilities.tools` is `{}` here — no `listChanged`, since
   the tool set is fixed at build time. `instructions` turned out to be the right
   home for the "provisioning is a human action" contract: it reaches the model
   even when the shipped skill was never installed.
3. **`tools/list` entry schema:** `{ name, title?, description, inputSchema,
   outputSchema?, annotations?, execution? }`. `inputSchema` **MUST** be a valid
   JSON Schema object and never `null`; a no-argument tool uses
   `{ type: "object", additionalProperties: false }`.
4. **`tools/call` result shape:** `{ content: [...], isError?,
   structuredContent? }`. Both options in the plan exist. **Chose text.**
   `structuredContent` must be a JSON *object*, and `fleet_list` returns an
   array — honoring it would mean wrapping that one result in an envelope (a new
   result shape, which Q2 forbids) or letting one tool answer unlike the other
   three.
   Error handling also split cleanly: unknown tools are protocol errors, while
   execution failures are `isError` results the model can act on.
5. **stdio framing:** newline-delimited JSON, and messages **MUST NOT** contain
   embedded newlines. The plan's highest-risk assumption was **correct**. A
   single-line `JSON.stringify` satisfies the rule for any payload, since it
   escapes newlines inside strings.
6. **Notifications:** the client MUST send `notifications/initialized`; messages
   without an `id` get no response at all. Also learned: a version the server
   does not support is **not** an error — the server answers with one it does
   support and lets the client decide.

## 3. Packaging — locked (Q1)

**Same package, `fleet mcp` subcommand. One bin, no new publish pipeline.**

MCP clients configure:

```
command: "fleet", args: ["mcp"]
# or, without a global install:
command: "npx", args: ["-y", "@switchyardhq/switchyard", "mcp"]
```

The deciding argument is version skew. Switchyard's MCP tools are a view onto
state the CLI mutates; a separately-versioned server could drift from the CLI
that created the worktrees and report stale semantics — and `fleet check`'s
verdict behavior already varies by git version, so semantic drift is a live bug
class here, not hypothetical. One package makes it structurally impossible.

`docs/deployment.md` needs no change under this option.

> **Note on wording:** Q1 said "a bin/entrypoint." This plan reads that as the
> `fleet mcp` subcommand on the existing bin (the option-A shape). If you meant a
> *second* `bin` entry (`switchyard-mcp`), that is a two-line `package.json`
> change plus a thin shim — say so and it folds into Task 4 at negligible cost.

## 4. Shipped Claude Code skill

Ship `skills/switchyard/SKILL.md` in the package (add `skills/` to `files` in
`package.json`, alongside `dist/` and `schema/`).

**Frontmatter**

```
name: switchyard
description: Use when working in a repo where multiple AI agents share one
  codebase — checking for collisions before editing and reading fleet state.
```

**Body instructs the agent to:**

1. **Establish position first** — call `fleet_list` to see whether a fleet exists
   and whether this agent already has a worktree.
2. **Work inside its own worktree, never the main checkout.** The entire
   isolation guarantee rests on this, and it is the failure mode that motivated
   the project.
3. **Call `fleet_check` before starting on a file, and again before requesting a
   merge** — and how to read verdicts: `will conflict` means stop and coordinate;
   `uncommitted edits` means another agent has unsaved work there that simulation
   could not see; a clean overlap is informational.
4. **Say explicitly that provisioning and merging are human actions in this
   release.** No spawn, merge, or remove tool exists. Without this the agent will
   hunt for one and then fall back to raw `git worktree` — exactly the
   uncoordinated behavior Switchyard exists to prevent. The skill should tell it
   to *ask* for `fleet spawn <name>` instead.
5. **Interpret `fleet_lock_status`** — a live lock means a human's mutation is in
   progress; wait and retry rather than working around it.

Point 4 is load-bearing under the reduced scope. The tools alone cannot convey
that checking is expected *before editing* rather than before merging, nor that
the missing mutations are deliberate — and that convention is the actual product.

**Open question:** whether the skill is auto-installed into `.claude/skills/` (a
`fleet init` responsibility, deferred to v0.4) or documented as a manual copy.
Recommend documenting for v0.3 — writing into a user's repo deserves its own
design pass.

## 6. Risks

### Dependencies

**No new runtime dependencies.** Footprint stays exactly `commander`,
`simple-git`, `chalk`, `cli-table3`.

The one candidate is `@modelcontextprotocol/sdk`, and it is **declined**. What it
would do: implement the JSON-RPC framing and MCP dispatch this plan hand-rolls in
Tasks 2–3 (~300 lines). Why not ours: the surface actually needed is small and
stable — one handshake, one listing method, one dispatch method, over
newline-delimited JSON — and Q2's read-only scope shrinks it further, since no
tool needs write semantics, progress reporting, or cancellation. Against that,
the SDK adds a transitive tree to a tool whose dependency minimalism is a
documented value, and this release exposes no advanced protocol features
(sampling, resources, prompts) that would justify it.

**Reassess if** the Task 0 spike reveals materially more protocol surface than
assumed, or if MCP versioning proves fast-moving enough that tracking it by hand
becomes a maintenance tax. That is a real possibility; the decision should be
revisited on evidence, not defended reflexively.

### Lock interaction

Materially smaller under Q2. No exposed tool calls `withLock`, so the
process-global reentrancy hazard in `src/lib/lock.ts:151` cannot trigger in
v0.3. Serialization is retained as forward-protection for v0.4's mutations, and
Task 4 asserts the server never creates `.fleet/lock`. **`lock.ts` is not
modified.**

### State schema

**No schema change.** `state.json` stays `version: 1`. v0.3 adds no persisted
state of its own — a direct consequence of deferring claims, whose
`.fleet/claims.json` is the only new state the earlier design introduced.

### Other

- **Stale reads.** The server must re-read state per call; agents spawn and merge
  underneath it. Any caching is a bug.
- **External mutation.** Branches deleted, worktrees removed, or `doctor --fix`
  run in another terminal must produce structured tool errors, never a crash that
  takes down the calling agent's session.
- **Testing a long-lived process.** Every existing test runs a command to
  completion. Subprocess tests must handle startup, request/response correlation,
  and teardown without hanging CI — budget for flakiness work, and ensure the
  server exits cleanly on stdin EOF.
- **Reduced-scope adoption risk.** A read-only surface may feel inert: the agent
  can see collisions but cannot act on them. If early use shows agents ignoring
  the tools because they can't do anything, that is the signal to promote
  `fleet_spawn` into v0.3.1 rather than waiting for v0.4.
